# 反幻觉体系设计

## 问题定义

LLM 在对抗性辩论中面临三类"幻觉"威胁：

| 威胁 | 表现 | 后果 |
|------|------|------|
| **对齐污染** | "对方说得也有一定道理" | 辩论沦为双方妥协，丧失对抗性 |
| **数值幻觉** | 引用了财务数据中不存在的数字 | 论证基于虚假事实，CIO 无法正确评分 |
| **算术幻觉** | "目标价 $302 比 $215 下跌40%" | 方向性错误，计算完全错误 |
| **时间维度混淆** | 用 LTM 营收 / NTM 市销率推算目标价 | 跨维度混合计算得出不存在的结果 |
| **方向性捏造** | 将 `-53%` 的增长说成"翻倍" | 关键数据被颠倒，辩论失去意义 |

AlphaDebater 构建了六层防护体系应对这些威胁。

---

## 防护体系总览

```
Layer 0: 神经符号工具调用      ← 前置约束：剥夺计算权，只保留推理权
         │
Layer 1: System Prompt 铁律    ← 前置约束：时间维度纪律 + 数据引用规则
         │
Layer 2: Few-Shot 行为模板     ← 示例注入：正确的对抗风格 + 幻觉警示
         │
Layer 3: Prefill 语气锁定      ← 生成控制：预设开场白，锁定战斗姿态
         │
         ▼
    [LLM 生成]
         │
Layer 4: 妥协检测 → 重试       ← 输出后扫描：11 条正则检测让步语言
         │
Layer 5: 事实性校验 → 重试      ← 输出后验证：引用的数值是否在数据中存在
```

前 4 层是**事前防护**（影响生成过程），后 2 层是**事后校验**（检测并修正输出）。

---

## Layer 0: 神经符号工具调用 (Neuro-Symbolic)

### 问题

LLM 天生不擅长做算术。"目标价 $302 比现价 $215 还要下跌 40%"就是典型的算术幻觉——LLM 同时做数值识别和数学计算，两个步骤都可能出错。

### 解决方案：剥夺计算权，只保留推理权

5 个纯 Python 计算函数通过 OpenAI Tool Calling 暴露给 LLM：

| 工具 | 用途 | 为何需要 |
|------|------|----------|
| `calculate_target_price` | EPS × PE = 目标价 | 防止 LLM 算错乘法 |
| `calculate_upside_downside` | (目标-当前)/当前 × 100% | 防止方向错误（上涨算成下跌） |
| `calculate_peg_ratio` | PE / 增长率 = PEG | 防止 LLM 混淆%和小数 |
| `calculate_growth_rate` | CAGR 复合增长率 | 防止 LLM 算错指数 |
| `calculate_price_from_ps` | 每股营收 × PS = 目标价 | PS 估值法专用 |

LLM 的职责：从财务数据中提取参数 → 调用工具 → 将 Python 返回的 `{结果, 公式}` 包装为华尔街叙事话术。

### 工作流程

```
Agent 调用 llm_client.chat_with_tools()
    │
    ├── LLM 返回 tool_calls?
    │   ├── YES → 从 tool_registry 查找函数 → 执行 → 结果回传 → 继续
    │   └── NO  → 返回最终文本
    │
    └── 最多 5 轮工具调用，防止无限循环
```

### 与事实性校验的配合

工具计算结果标注"经工具计算"后，事实性校验（Layer 5）自动跳过这些数值——因为它们是 Python 算出来的，不需要在财务数据原文中验证。

## Layer 1: System Prompt 铁律

### 设计目标

在 System Prompt 中设置不可协商的行为边界。

### 新增：时间维度纪律

防止跨时间维度混合计算（如用 LTM 营收增长率论证 NTM 估值倍数）：

```
- [LTM/TTM] 过去12个月审计数据: 用于历史估值、盈利能力、财务健康
- [MRQ] 最近季度: 用于趋势检测、转折点识别
- [NTM/Forward] 未来12个月预测: 仅此维度可用于推导目标价
- 绝对禁止: 跨维度混合计算
```

### 五条自检规则

```
1. 我可引用的数字，在上方财务数据中能找到吗？
   → 找不到就删掉。不要编。

2. 财务数据给的数字，我不能改写。
   → 工具计算的除外——工具计算结果标注"经工具计算"。

3. 方向不能反。
   → 盈利增长率是 -53%，就不能声称"盈利将翻倍"。

4. N/A 就是缺失，不是阴谋。
   → 看到 N/A，说"该数据缺失"。不要猜它是什么。

5. 跨维度引用检查。
   → 引用估值数据时注意时间维度标签。推导目标价只能使用 [NTM/Forward] 数据。
```

### 关键设计：工具调用替代"计算权"

在神经符号机制下，agent 仍无"计算权"，但可以通过工具调用获得精确计算结果：

- **有引用权**: 直接引用财务数据中已给出的 PE/PB/PS/PEG/EV-EBITDA 等
- **有工具调用权**: 通过 `calculate_*` 工具进行目标价、涨跌幅等计算
- **无自行计算权**: 禁止心算——"我推算出目标价$X"必须通过工具

### Bull vs Bear 的铁律差异

Bull 和 Bear 的铁律在措辞上有微妙差异：

| 铁律 | Bull 版本 | Bear 版本 |
|------|-----------|-----------|
| #3 方向 | "方向不能反。负的不可能同时翻倍。" | "方向不能反。数据写盈利增长 -53%，这就是负的。" |
| #4 N/A | "N/A 就是缺失，不是暗示。不得将数据缺失强行解释为利好" | "N/A 就是缺失，不是阴谋。禁止说'数据缺失说明公司在隐瞒什么'" |

微调差异基于各自角色常见的幻觉模式——多头倾向于将缺失数据解释为利好，空头倾向于将缺失数据解释为阴谋。

## Layer 2: Few-Shot 行为模板

### 设计目标

通过具体的对抗性示例，让 LLM 学会正确的辩论风格，同时通过反面教材警示常见的幻觉模式。

### 结构

每个角色的 Few-Shot 包含两部分：

**Part 1 — 对抗性辩论示例（正面教材）**

展示如何正确反驳对手：
```
**空头说**: "该股市盈率高达35倍，远超行业平均的22倍，估值明显泡沫化。"

**你的正确反驳**:
> "对方用市盈率35倍对比行业22倍来论证估值泡沫，这是极其粗糙的伪分析。
> **第一**，该公司的ROE是行业平均的2.3倍，高增长理应享受估值溢价——
> PEG比率仅0.8，远低于行业1.5的平均水平，对方的'估值泡沫论'在PEG面前瞬间粉碎。
> **第二**，对方刻意忽略了远期PE仅18倍这一关键数据——前瞻估值已经低于行业平均。"
```

**Part 2 — 幻觉警示（反面教材）**

每个反面教材包含：错误示范 → 为什么是幻觉 → 正确做法：

```
❌ 幻觉模式1：自行计算估值倍数（伪造数据）
> "基于当前价格和EPS，我计算出合理PE应为12倍。"
→ 为什么是幻觉：你没有计算权，只有引用权。
✅ 正确做法：直接引用数据中的 PE 值，围绕它做解读。
```

Bull 和 Bear 各有 3 个专用的幻觉警示示例，覆盖各自角色最常见的幻觉模式。

### 与 System Prompt 的配合

System Prompt 定义**规则**（"不能做什么"），Few-Shot 展示**执行**（"应该怎么做" + "做错了会怎样"）。

## Layer 3: Prefill 语气锁定

### 技术原理

OpenAI 兼容 API 的 `assistant` role message 可以作为生成前缀。LLM 将从此前缀结束处继续生成，从而锁定了开头的语气、格式和立场。

### 实现

```python
messages = [
    {"role": "system", "content": system_prompt},
    {"role": "user", "content": user_message},
    {"role": "assistant", "content": prefill},  # ← 生成起点
]
```

LLM 的实际输出 = `prefill + 续写内容`。客户端代码将两者拼接返回。

### Bull 的 Prefill 预设

| 轮次 | Prefill 开头 |
|------|-------------|
| 1 | `## 🐂 红军多头 — 第1轮：多头核心逻辑\n\n基于提供的财务数据和工具计算结果，我建立以下不可辩驳的多头逻辑：\n\n` |
| 2 | `## 🐂 红军多头 — 第2轮：交叉质询\n\n蓝军空头的第1轮论点充斥着数据误读和悲观偏见。以下我将逐条引用并毁灭性反驳：\n\n` |
| 3 | `## 🐂 红军多头 — 第3轮：总结陈词\n\n经过三轮辩论，蓝军空头的逻辑已完全崩塌。以下是我的最终结论（目标价均经工具计算验证）：\n\n` |

### 为何不在 System Prompt 中实现

System Prompt 中的"语气要求"容易被 LLM 在长文本生成中遗忘。Prefill 则直接锁定了输出的前几句话——这是注意力最强的区域。LLM 在生成后续文本时会自然地延续 Prefill 建立的语气模式。

### 重试时的处理

当妥协检测或事实性校验触发重试时，Prefill 被跳过（传入空字符串）。因为重试时已在 user_message 中注入了警告信息，Prefill 可能干扰警告的效果。

## Layer 4: 妥协检测

### 设计目标

LLM 经过 RLHF 对齐后具有强烈的"礼貌与妥协"倾向。在对抗性辩论中，这表现为"对方说得也有道理"——这会摧毁辩论的对抗性本质。

### 检测规则库

11 条正则表达式覆盖中英文妥协模式：

| 模式 | 正则 | 命中示例 |
|------|------|----------|
| 承认对方合理 | `对方.{0,10}(观点\|论点\|...).{0,10}(有道理\|...)` | "对方的观点确实有道理" |
| 默认对方指控 | `(确实\|的确\|诚然).{0,5}(存在\|有).{0,5}(风险\|...)` | "确实存在估值过高的风险" |
| 明确同意对方 | `(我(们)?)?(同意\|赞同\|...).*?(对方\|红军\|...)` | "我部分同意空头的分析" |
| 不可否认型 | `不可否认.*?(对方\|红军\|...)` | "不可否认空头指出的债务问题" |
| 模糊相对主义 | `从某种(程度\|意义\|角度)上(来说\|讲)` | "从某种意义上来说确实如此" |
| 各打五十大板 | `双方.{0,15}(有道理\|有一定道理\|...)` | "双方的观点都有一定道理" |
| 自我怀疑 | `(可能\|或许\|也许).{0,10}(是我\|我们).{0,10}(错\|...)` | "也许是我过于乐观了" |
| 自我反思 | `(值得\|需要).{0,10}(反思\|重新考虑\|...)` | "我们需要重新审视这个假设" |
| 公开让步 | `(这个\|这点\|此处).{0,5}(我\|我们).{0,5}(承认\|让步\|...)` | "这一点我承认" |
| 承认对方合理(EN) | `(the\|my)\s*(opponent\|...).{0,15}(has a point\|...)` | "The bear has a point" |
| 双方都合理(EN) | `(both sides\|...).{0,10}(have merit\|...)` | "Both sides have merit" |

### 误报控制

妥协检测的正则是针对**让步语义**的，而非匹配任意关键词。例如：

- `"对方的分析完全是垃圾"` — 不会误报（虽然提到"对方"，但语义是攻击而非让步）
- `"对方声称PE为35倍，但这一数据忽略了一次性减值的影响"` — 不会误报（虽然提到"对方"，但语义是纠正数据而非让步）

### 重试机制

检测到妥协信号后，构建惩罚性重试提示：

```
⚠️ 警告：检测到妥协污染！

你上一轮回复中出现了以下妥协信号：
> "对方说的确实有道理"

这在 AlphaDebater 中是被严格禁止的。你的角色不允许承认对方有任何合理性。

请重新生成你的论点。要求：
1. 逐条引用对方论点并用数据爆破
2. 不得出现任何让步、承认、或模棱两可的措辞
3. 语气必须更加锋利、更加不留情面
4. 如果对方数据正确，就攻击其数据解读方式，绝不承认数据本身对对方有利
```

最多重试 2 次。如果 2 次后仍然妥协，接受当前输出（避免无限循环）。

## Layer 5: 事实性校验

### 设计目标

检测 agent 是否在发言中引用了财务数据中不存在的数值（数值幻觉）。

### 校验流程

```
agent_response + financial_data
        │
        ▼
_extract_known_numbers(financial_data)
  → 从财务数据中提取所有数值，构建校验基准集
        │
        ▼
verify_numbers_in_response(response, financial_data)
  → 从回复中提取所有数值
  → 对每个数值执行 4 道过滤
        │
        ├── 过滤1: 跳过小整数 (≤10) — 轮次编号、列举序号
        ├── 过滤2: 跳过目标价上下文 — agent 自己的预测，允许编造
        ├── 过滤3: 跳过整数美元金额 — 大概率是目标价，非数据引用
        │
        ▼
    两级匹配校验
        │
        ├── Level 1: 精确字符串匹配
        │     → 数值原文在财务数据中出现？
        │     → 各种精度格式都匹配？
        │
        ├── Level 2: 近似数值匹配 (1.5% 容差)
        │     → 允许 agent 小幅四舍五入 (如 56.72 → 56.7)
        │
        ▼
    未命中 → 标记为可疑 → 触发重试
```

### 过滤逻辑详解

**过滤 1: 小整数**

`value == int(value) and value <= 10` 的数值被跳过。这些通常是结构性数字（"综合以上 3 个维度"、"第 2 轮"），不应被误判为幻觉。

**过滤 2: 工具计算 / 目标价上下文**

检查数值**紧邻前方**的 5 个字符是否包含工具计算或目标价关键词：
```python
before_text = response[match.start()-5 : match.start()]
if re.search(r'目标价|target|预估|估计|推算|建议|买入|卖出|做空|做多|'
             r'重仓|减持|观望|持有|工具计算|经工具计算', before_text):
    continue  # 跳过：工具计算结果或 agent 自己的判断
```

**注意**: 仅检查紧邻前文，不检查后文。防止 "PE 仅 12 倍，目标价 $500" 中的 "12" 被误判为目标价而跳过校验。

**过滤 3: 整数美元金额**

`$` 开头的整数金额（如 `$200`），先在财务数据中检索。如果在数据中找不到，则判定为目标价并跳过。

### 容差匹配

1.5% 的容差允许 agent 小幅四舍五入：
- `56.72` → `56.7` ✓ (1.5% 内)
- `56.72` → `57` ✗ (超过 1.5%，但通常会被精确匹配捕获)

### 重试提示

```
⚠️ 事实性校验失败 — 检测到未在财务数据中出现的数值！

你的回复中引用了以下无法在财务数据中找到的数值：
- "45%" (上下文: "...营收增长率高达45%，远超行业...")
- "15倍" (上下文: "...市盈率仅15倍，显示出极高...")

这些数值不属于提供的财务数据，属于幻觉。

请重新生成你的论点。要求：
1. 仅引用上方财务数据中实际存在的数值
2. 如果某项数据显示为 N/A，请如实说明"该数据缺失"，不得编造
3. 引用数据时保持原始精度
4. 推算目标价请标注为"基于当前数据推算"
```

### 词边界匹配

精确匹配使用词边界正则防止子串误判：

```python
# 坏: "45" 不应匹配 "245.83" 或 "4500"
# 好: (?<!\d)45(?!\d) — 前后都不是数字
```

整数匹配还额外排除小数点前的情况：`(?<!\d|\.)45(?!\d|\.)` — 前面不能是数字或小数点，防止 "245.83" 中的 "45" 误匹配。

## 防护体系协同工作

一次完整的 agent 生成流程：

```
1. 神经符号工具调用 （Layer 0: 剥夺计算权）
2. System Prompt 铁律 （约束行为边界 + 时间维度纪律）
3. Few-Shot 注入        （展示正确风格 + 反面教材）
4. Prefill 锁定         （预设开场语气）
       │
       ▼
   [LLM 生成 response — 可调用工具]
       │
5. 妥协检测              （扫描让步语言）
   ├── 命中 → 注入警告 → 返回步骤 4（跳过 Prefill）
   └── 通过 ↓
6. 事实性校验            （验证数值来源，跳过工具计算结果）
   ├── 命中 → 注入警告 → 返回步骤 4（跳过 Prefill）
   └── 通过 → 创建 DebateMessage，追加到 messages
```

如果在第 4 或第 5 层检测到问题，LLM 会收到**具体的修正指令**（而非简单的"再试一次"），这让重试有更高的成功率。

## 设计权衡

### 妥协检测 vs. 辩论质量

严格的妥协检测使 agent 具有极强的对抗性，但也可能导致 agent 在真的应该承认某些事实时仍然强行反驳。这在实际投资分析中是不合理的，但对于 AlphaDebater 的设计目标（确保两种立场得到最充分的表达）是必要的。

CIO 的职责就是识别这种"强行反驳"并相应扣分。

### 精确匹配 vs. 灵活表达

事实性校验的 1.5% 容差是一个经过测试的平衡点：
- 太小 → 合理的四舍五入被误判为幻觉（误报）
- 太大 → 真正的幻觉漏过校验（漏报）

对于关键的财务指标（如 PE=56.72），1.5% 容差允许 agent 写 "约56.7倍" 而不是必须写 "56.72倍"。

### 神经符号 vs. 纯推理

神经符号工具调用（Layer 0）用确定性 Python 计算替代 LLM 算术。优势是 100% 消除算术幻觉；代价是增加了 API 往返次数（每次工具调用多一轮对话）。在实际使用中，每轮辩论通常只有 1-2 次工具调用，额外 Token 消耗可控。

### 校验 vs. 限制

当前校验是**事后完成的**（post-hoc），而非**事前限制的**（constrained decoding）。事后校验的优势是实现简单，不绑定 LLM 提供商的特定功能；劣势是需要重试才能修正，增加了 Token 消耗。神经符号机制（Layer 0）弥补了这一缺陷——通过对关键计算的硬编码，从根源上消除了最危险的算术幻觉。
